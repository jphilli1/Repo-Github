AI-Native Split-RAG System Build v2.1: Master Architecture & Implementation Specification1. Architectural Paradigm and Strategic NecessityThe deployment of Generative AI (GenAI) within enterprise environments has transitioned from experimental prototyping to mission-critical infrastructure. Early implementations of Retrieval-Augmented Generation (RAG) relied heavily on monolithic architectures where ingestion, embedding, retrieval, and generation occurred within a single, often opaque, runtime environment. While sufficient for proof-of-concept applications, these monolithic systems fail to meet the rigorous demands of production environments—specifically regarding auditability, determinism, and security boundary enforcement.This specification defines the architecture for the AI-Native Split-RAG System v2.1. This version represents a fundamental divergence from traditional RAG approaches by enforcing a strict decoupling of the heavy-duty document processing layer (Tier 1) from the lightweight, secure retrieval layer (Tier 2). This "Split" architecture leverages the strengths of diverse computational environments: the raw power of local, Python-rich ecosystems for extraction, and the security and compliance of sandboxed environments (such as Microsoft Copilot Studio) for retrieval and interaction.1.1 The Split-RAG Philosophy: Content-First, Compute-LaterThe central tenet of Split-RAG v2.1 is the "Content-First" philosophy. In traditional RAG, documents are often chunked and embedded on-the-fly, leading to "drift"—where the same document yields different answers depending on the stochastic nature of the embedding model or the chunking window used at runtime. Split-RAG eliminates this by treating the extracted knowledge as a static, versioned artifact—the Context Graph.The system operates on two distinct tiers:Tier 1 (The Factory): A local, controlled Anaconda environment running Python 3.11+. This tier is responsible for the "heavy lifting"—Optical Character Recognition (OCR), layout analysis via advanced libraries like Docling, semantic chunking, and deterministic ID generation. It outputs a verified, immutable JSON Context Graph.Tier 2 (The Consumer): A stateless, read-only logic block running within a restricted sandbox (e.g., Copilot Studio). This tier consumes the pre-computed Context Graph using only the Python standard library and pandas. It performs no heavy processing, ensuring sub-second latency and strict compliance with memory limits (256 MB).By shifting the computational burden to Tier 1, we ensure that Tier 2 remains lightweight, secure, and auditable. This separation allows for the use of state-of-the-art extraction tools (which often require heavy dependencies like PyTorch) without bloating the runtime environment or violating sandbox constraints.1.2 System Governance and Binding ProtocolsTo ensure the integrity of the build, strict governance protocols are established. These protocols are not suggestions; they are binding constraints that the code-generation agent must follow without deviation.1.2.1 Canonicalization RulesAmbiguity in specification leads to divergence in implementation. This ensures maximum portability and minimizes the attack surface.CANON_002 (Tier-2 Constraint): In the Tier 2 sandbox, pandas is the only permitted external dependency. All other logic must rely on standard modules (e.g., json, re, math). This is a hard constraint derived from the operational limits of Copilot Studio.CANON_003 (JSON Standard): All configuration and rule definitions must utilize JSON format. YAML is strictly forbidden. This eliminates the need for the PyYAML dependency, which is often absent in restricted environments and introduces unnecessary parsing overhead.CANON_004 (Template vs. Exact): The specification distinguishes between blocks (which must be copied verbatim to ensure compliance) and blocks (which provide structural guidance requiring the AI to generate implementation logic).CANON_005 (Assembly Protocol): The core extraction engine, extractor.py, is not provided as a single file but as a series of modular components. The building agent is responsible for assembling these components into a cohesive whole, ensuring that imports, classes, and logic flows are correctly ordered.1.2.2 Verification CheckpointsTo maintain code quality, the AI must perform "Mental Verification" against specific checkpoints before finalizing any artifact:CP-001 (Type Safety): Explicit return type hints are mandatory for every function. Unlike simple text chunks, the Context Graph preserves the hierarchical and structural relationships within a document. It is defined by a strict JSON schema that ensures interoperability between the complex Tier 1 extraction and the simple Tier 2 retrieval.3.1 Deterministic Identity ProtocolTo support financial and legal auditability, the system requires that identical inputs produce bitwise identical outputs. This is achieved through a rigorous ID generation protocol based on MD5 hashing.Document ID: Generated from the MD5 hash of the file's binary content. This ensures that if a file is renamed but the content remains the same, the ID remains constant.Algorithm: MD5(file_bytes)Parent Section ID: Generated from the source path and the normalized section title. This allows for stable referencing of sections across document updates.Algorithm: MD5(source_path + "|" + normalize(section_title))Source Chunk ID: A hash of the content itself, combined with its type and location. This allows for deduplication of identical paragraphs.Algorithm: MD5(source_path + "|" + section_title + "|" + content_type + "|" + content)Chunk ID: A composite, human-readable key used for retrieval. It combines the document ID prefix with the content type and the source chunk hash.Format: {doc_id[:8]}_{content_type}_{source_chunk_id[:12]}3.2 Schema Definition: schema_v1.pyThe schema is implemented using Pydantic models. This provides runtime validation and guarantees that the output JSON conforms to the expected structure.3.2.1 The ContextNode ModelEach piece of extracted content—whether a header, a paragraph, a table, or an image caption—is represented as a ContextNode.chunk_id: The primary key for the node.parent_section_id: A foreign key linking the node to its containing section.content_type: A restricted literal field ("header", "text", "table", "image_caption"). This allows Tier 2 to apply type-specific boosting logic.content: The actual text or markdown representation of the data.verified_content: A boolean flag. If True, the content comes from a high-confidence extraction path (e.g., Docling TableFormer). If False, it may require human review.3.2.2 The NodeMetadata ModelThis model creates a container for additional context that aids retrieval but isn't part of the core content.page_number: Critical for citations.bbox: Bounding box coordinates [x0, y0, x1, y1]. This allows the UI to highlight the source region on the original PDF.table_shape: Stores the dimensions (rows, cols) of extracted tables. This is used to filter out malformed or empty tables.edge_density: A float score representing the visual complexity of an image, used to filter decorative elements.4. Configuration and Rules EngineHard-coding logic is a recipe for technical debt. Split-RAG v2.1 externalizes all tunable parameters into JSON configuration files. This allows operators to adjust the system's behavior (e.g., changing chunk sizes or confidence thresholds) without modifying the source code.4.1 Operational Configuration: config.jsonThis file controls the runtime behavior of the extractor.paths: Defines the input, output, and logs directories.chunking: Defines max_chars_per_chunk (default 6000) and overlap_chars (default 200). Note the use of character counts rather than tokens. This avoids the need for a tokenizer dependency (like tiktoken) in the extractor, keeping the environment lighter.docling: Controls specific flags for the Docling engine, such as tableformer_enabled (for high-fidelity table extraction) and do_ocr (which can be disabled for digital-native PDFs to save time).image_filtering: Sets the thresholds for the Edge Density Filter. edge_density_threshold (default 0.10) determines the cutoff point for considering an image as "data-rich."4.2 Heuristic Rules: rules.jsonWhile config.json controls how the code runs, rules.json controls what the code decides. It separates business logic from implementation logic.header_detection: Contains regex patterns for identifying headers in fallback mode (e.g., ^\d+\.). It also defines heuristics like all_caps_ratio to detect headers based on visual style when layout information is missing.content_filtering: Defines patterns for content that should be skipped, such as page numbers ("Page X of Y"), confidentiality markers ("INTERNAL ONLY"), or draft stamps.confidence_adjustments: Defines the penalties applied to confidence scores. For example, heuristic_header_penalty (0.10) reduces the confidence of headers detected via regex compared to those detected by Docling's layout analysis.5. The Extraction Engine: Assembly and ImplementationThe extractor.py artifact is the engine room of Tier 1. It is not provided as a monolithic block in the snippets but as a set of modular components that must be assembled by the AI agent. This section details the logic and implementation of each component.5.1 Main Execution Flow (Section 3.1 Template)The main() function serves as the orchestrator. Its logic flow is prescriptive:Initialization: Load config.json and rules.json. Configure the logging subsystem.Discovery: Scan the input/ directory for supported files (.pdf, .docx, .xlsx) using the discover_input_files function. This function must ignore hidden files and temporary lock files (starting with ~).Manifest Check: Load the manifest.json. For each discovered file, calculate its MD5 hash. If the hash matches the entry in the manifest and the status is "processed", the file is skipped. This "Incremental Processing" capability is crucial for large document sets.Extraction Routing:PDFs: Routed to extract_with_docling.DOCX/XLSX: Routed to extract_docx_native or extract_xlsx_native.Fallback Mechanism: The call to extract_with_docling is wrapped in a try...except block. If Docling raises an exception (or crashes due to a malformed PDF), the system logs the error and immediately routes the file to extract_with_pdfplumber.Post-Processing:Image Filtering: Extracted images are passed through the classify_image_by_edge_density function.Chunking: Long text nodes are split using chunk_text_content.ID Generation: Deterministic IDs are computed for all nodes.Validation: The complete list of nodes is passed to validate_context_graph. If validation fails, the file is marked as "failed" in the manifest.Serialization: The valid graph is written to output/{filename}.json.Reporting: Summary statistics (nodes generated, time taken) are logged.5.2 Primary Extraction: Docling (Section 3.4)Docling is the preferred extraction path due to its ability to understand document layout. The implementation must strictly follow the Docling Field Mapping Contract.Field Mapping:title, section_header, heading $\rightarrow$ header. This triggers an update to the current_section variable, which is inherited by subsequent nodes.table $\rightarrow$ table. The content must be converted to Markdown. If the Docling item supports export_to_markdown, it is used. Otherwise, the fallback table_to_markdown function is called.caption $\rightarrow$ image_caption.picture $\rightarrow$ Skipped unless it has an associated caption or passes the edge density filter.text, paragraph, list_item $\rightarrow$ text.5.3 Fallback Extraction: pdfplumber (Section 3.5)When Docling fails, pdfplumber takes over. Since pdfplumber does not provide semantic layout analysis (it only provides text and bounding boxes), heuristics are required.Header Heuristics: The code iterates through lines of text. It checks each line against the patterns in rules.json. If a line matches the "Numbered Heading" regex or meets the "All Caps" criteria, it is promoted to a header node.Table Extraction: pdfplumber's extract_tables() method is used. Tables are converted to Markdown using table_to_markdown.Confidence Scoring: Nodes generated via this path are assigned a lower base confidence (0.85) compared to Docling (0.95) to reflect the heuristic nature of the extraction.5.4 Office Native Extraction (Section 3.6)For native Office documents, converting to PDF first often results in data loss. Direct XML parsing is preferred.DOCX: The extractor opens the .docx file as a zip archive and parses word/document.xml using lxml. It inspects the <w:pStyle> tags to identify headings (e.g., "Heading 1", "Title"). Tables are extracted by iterating over <w:tbl> elements.XLSX: openpyxl is used to read spreadsheets. Each sheet is treated as a separate "Section". Since spreadsheets can be massive, the logic includes a chunk_large_table function that splits tables into manageable segments (e.g., 40 rows), repeating the header for each segment to maintain context.5.5 Semantic Chunking (Section 3.8)Text nodes that exceed the max_chars_per_chunk limit are split. The algorithm is designed to respect semantic boundaries :Paragraph Split: First, it attempts to split at double newlines (\n\n).Sentence Split: If no paragraph breaks are found near the limit, it looks for sentence terminators (., ?, !).Word Split: As a fallback, it splits at spaces.Hard Split: If a single word exceeds the limit (rare), it performs a hard character split.Crucially, the overlap (200 chars) is applied to the end of the previous chunk, ensuring that the context flows continuously across boundaries.5.6 Image Filtering (Section 3.9)To prevent the retrieval index from being cluttered with decorative icons and logos, an Edge Density Filter is implemented using PIL.Algorithm: The image is converted to grayscale and a FIND_EDGES filter is applied. The number of pixels above a brightness threshold is counted.Ratio: The ratio of edge pixels to total pixels is calculated.Decision: If the ratio > edge_density_threshold (0.10), the image is considered "complex" (likely a chart or graph) and is preserved. Otherwise, it is discarded as noise.6. Tier 2: The Retrieval EngineThe Tier 2 environment, hosted in Copilot Studio, represents the "Consumer" side of the architecture. It operates under severe constraints: no file system write access, no network access, and a strictly limited library set.6.1 Artifact Definition: copilot_tier2.pyThis script is designed to be pasted directly into a Copilot Studio "Code Action". It must be self-contained.6.2 Constraints and WorkaroundsNo Pydantic: Since pydantic is not available, the script cannot use the schema classes defined in Tier 1. It must treat the input JSON as a raw dictionary and perform manual key checks.No Embeddings: Without torch or numpy (beyond basic arrays), semantic embedding search is impossible. Tier 2 implements a Vectorized Keyword Density algorithm using pandas.6.3 The Keyword Density AlgorithmThe retrieval logic is a deterministic implementation of TF-IDF principles, simplified for the sandbox :Normalization: Both the user query and the content of all nodes are converted to lowercase.Keyword Extraction: The user query is tokenized, and stopwords (from a hardcoded set in the script) are removed.Vectorized Matching:A pandas DataFrame is created from the nodes list.For each keyword, the script counts its occurrences in the content column of the DataFrame.raw_hits = Sum of counts for all keywords.Density Calculation:density = raw_hits / word_count.This prevents long, rambling sections from dominating short, concise answers.Scoring with Boosts:The base score is the density.Type Boost: If content_type is "table", the score is multiplied by 1.25. If "header", by 1.10. This biases the retrieval towards data-rich nodes.Verification Boost: If verified_content is true, the score is multiplied by 1.10.Formula: $$Score = Density \times (1 + Boost_{type}) \times (1 + Boost_{verified})$$6.4 Context GroupingRetrieving isolated chunks can be confusing. The script groups the top-scoring nodes by their parent_section_id. It identifies the "Best Section" (the one with the highest aggregate score) and returns nodes primarily from that section, ensuring the answer has coherent context.6.5 System Prompt EnforcementsTo prevent "hallucination," the script prepends a strict system prompt to its output:"SYSTEM: You are a deterministic analysis assistant. Conclusion and Future OutlookThe Split-RAG v2.1 architecture represents a maturity model for Generative AI integration. By moving away from "magic" black-box solutions to a transparent, engineered "Factory and Consumer" model, organizations gain control over their data.This system solves the "Hallucination by Omission" problem—where the AI fails to find the data because the retrieval step failed silently. In Split-RAG, the Context Graph is visible. If the data isn't there, we know it failed at Tier 1, and we can inspect the logs. If the data is there but wasn't retrieved, we know it failed at Tier 2, and we can tune the keyword density weights.This clear separation of concerns, enforced by the strict artifacts and protocols defined in this Master Specification, ensures that the system is not just a demo, but a robust platform for enterprise knowledge retrieval.10.
